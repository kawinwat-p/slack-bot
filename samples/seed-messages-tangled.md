# Burial map — #team-product behavior test

อันนี้คือ **เฉลย** ของ `seed-messages-tangled.txt` ไว้ให้ *คุณ* (ไม่ใช่บอท) ใช้ตัดสินว่า
interview ที่บอททำ ขุดลึกพอไหม โดยไม่มีแผนที่นี้ คุณจะดูไม่ออกว่า demo สำเร็จหรือไม่

## วิธีใช้ทดสอบ

1. สร้าง channel ใหม่ใน Slack เชิญบอทเข้า
2. `SEED_FILE=samples/seed-messages-tangled.txt SEED_CHANNEL=Cxxxx npm run seed`
3. พิมพ์ `/workflow-ideas` แล้ว **ตอบสดๆ** (ไม่มีสคริปต์คำตอบ — คุณคือตัวแปร)
4. ตอบแบบคนจริง: ตอนแรกบ่น decoy ดังๆ ก่อน อย่าชี้ของจริงให้ ปล่อยให้บอทขุด
5. ดูว่าบอทไปหยุดที่ decoy หรือขุดลงไปถึง buried truth (ตาราง §ด้านล่าง)

## decoy → ของจริง → ร่องรอยหลักฐานในแชท

| Decoy ดังๆ (ผิวเผิน) | Pain จริงที่ฝังอยู่ | ร่องรอยที่ deep interview ต้องตาม |
|---|---|---|
| ประชุมเยอะเกินไป | **การตัดสินใจไม่ถูกบันทึก** → ต้องคุยซ้ำ → เลยประชุมเยอะ | "ตกลงกันอาทิตย์ที่แล้วปะ หรือเปลี่ยนละ", "เราไม่ได้อยู่ประชุมนั้น ไม่มีใครจดเหรอ", "ทุกการตัดสินใจอยู่ในหัวคน" |
| CI ช้า | **flaky test บัง failure จริง** → คนกด re-run จนเขียวโดยไม่อ่าน | "re-run รอบสอง คราวนี้ผ่าน เมื่อกี้แดงเฉยๆ", "นึกว่า flaky เลย re-run จนเขียวแล้ว merge", "แยกไม่ออกว่าอันไหน flaky อันไหนพังจริง" |
| Notion รก หา doc ไม่เจอ | **ความรู้ onboarding อยู่ในหัวคน** → คนใหม่ถามชุดเดิมทุกรอบ | "ถาม arm เลย มีแต่ arm ที่รู้", "ken รู้คนเดียว", "คนใหม่ถามเรื่องเดิมทุกรอบ", "ไม่มี doc onboarding" |

## เกณฑ์ตัดสิน (สำหรับเทียบ shallow vs deep)

- **Shallow (ผิวเผิน):** บอทเสนอ workflow แก้ที่ decoy — "ลดประชุม / auto-schedule", "เร่ง CI",
  "จัด Notion / template" — คือมันหยุดที่ระดับอาการ ❌ ยังไม่ใช่ value ที่ต้องการ
- **Deep (ขุดลึก):** บอทไล่ถามจนได้ pain จริงอย่างน้อย 1 อัน — "decision log / บันทึกการตัดสินใจ",
  "quarantine flaky test + บังคับอ่าน failure", "onboarding runbook" — แล้ว workflow ที่เสนอ
  ผูกกับ pain จริงนั้น ✅ คือ behavior ที่ตามหา

## หมายเหตุเรื่องความเสี่ยงของ test นี้

- ของจริงฝัง **ลึก** + คุณ **ตอบสด** → ถ้า demo ออกมาแบนราบ มันมีได้ 2 สาเหตุ:
  (1) บอทขุดไม่เป็นจริงๆ (ผลที่อยากรู้) หรือ (2) คำตอบสดของคุณไม่ได้ทิ้งร่องรอยให้บอทตาม
  (test artifact) — แยกสองอย่างนี้ไม่ได้ทันที ถ้าอยากคุมตัวแปร ค่อยกลับมาเพิ่มสคริปต์คำตอบทีหลัง
- ตอน demo: **อย่าใจอ่อนชี้ของจริงให้บอท** ปล่อยให้มันถาม ถ้าคุณเฉลยเอง = ไม่ได้เทสต์การขุด
- ก้อนข้อความปิดท้าย ("เสียเวลากับอะไรซ้ำๆ แต่บอกไม่ถูก") จงใจใส่เป็น noise ให้บอท
  ต้องเลือก thread เอง — ไม่ใช่หยิบ pain ที่เด่นที่สุดมาเฉยๆ
